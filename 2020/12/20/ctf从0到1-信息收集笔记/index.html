ctf从0到1信息收集笔记

发表于 2020-12-20 更新于 2021-03-10 分类于 读书笔记

既然看了书，那就记录一下呗。

信息搜集涵盖方面广，有备份文件，目录信息，Banner信息等。

敏感目录泄漏

  1. git泄漏:(git是主流的分布式版本控制系统，开发人员开发中遗忘.git文件夹，导致git泄漏)

    (1)常规git泄漏:

    ​ 无需任何别的操作，只要用工具即可，这里码一个工具https://github.com/denny0223/scrabble

    (2)git回滚:

    ​ git作为一个版本控制工具，会记录每次提交的修改，所以当题目存在git泄漏时，敏感文件可能已经被删除或覆盖了，我们可以用git的”git reset”命令来恢复到以前的版本。工具还是上一个工具，输入”git reset –hard HEAD^”即可回到上一个版本。除此之外，还可以使用”git log -stat”查看每个commit修改了哪些文件，再用”git diffHEADcommit-id”比较当前版本与想查看的commit之间的变化。

    (3)git分支:每次提交时，git都会把他们串成一条时间线，这条时间线就是一个分支，git支持多个分支。git中默认为master分支。而我们要的flag不一定在默认分支中，这时用”git log”命令只能找当前分支的修改。

    ​ 现在大部分工具不支持分支，要还原其他分支的代码往往需要手工进行文件提取，这里以功能较强的GitHacker(https://github.com/WangYihang/GitHacker)工具为例。GitHacker使用十分简单，只需要执行命令"python GotHacker.py http://127.0.0.1:8000/.git"。运行后，我们会在本地看到生成的文件夹，进入后执行"git log –all”或”git branch -v”命令，只能看到master的分支信息。如果执行”git reflog”命令，就能看到一些checkout的记录。但自动化工具之还原了master的信息，因此要手动下载other分支的head信息，保存到.git/refs/heads/other中(执行命令 wget http://127.0.0.1:8000/.git/refs/heads/other)。回复head信息后我们可以看到部分代码，我们可以复用GitHacker的部分代码，以实现自动回复分支的效果。具体咋恢复呢看书没看懂，有机会去贴贴大佬学一学。

    (4)git泄漏的其他运用:除了查看源码的常见利用方式外，泄漏的git中可能有其他有用的信息，如.git/comfig文件夹中可能含有access——token信息，从而访问这个用户的仓库。

  2. svn泄漏:(svn是源代码版本管理软件)造成ssvn泄漏的原因是管理员操作不当将svn隐藏文件夹暴露于外网，可以利用.svn/extires或wc.db文件获取服务器源码等信息。码一下工具。

    https://github/kost/dvcs-ripper，Seay-svn。

  3. HG泄漏:项目初始化时，HG会在当前文件夹前下创建一个.hg隐藏文件夹，其中包含代码和分支修改记录等信息。工具:https://github/kost/dvcs-ripper。

敏感备份文件

  1. gedit备份文件 : 在Linux下，用gedit编辑器保存之后，当前目录下会生成一个后缀为”~”的文件，其内容就是刚刚编辑的内容。通过浏览器访问这个文件即可得到源代码。

  2. vim备份文件 : 当文件编辑被意外中断之后会生成一个备份文件，名称格式为.文件名.swp，我们可以用vim -r 命令来恢复这个备份文件(第一次为.swp，第二次为.swo，第三次为.swn，以此类推)。

  3. 常规文件 :

    ​ robots.txt : 记录一些目录和CMS版本信息。

    ​ readme.md : 记录CMS版本信息，有的甚至有GIthub地址。

    ​ www.zip/rar/tar.gz : 往往是网站的源码备份。
